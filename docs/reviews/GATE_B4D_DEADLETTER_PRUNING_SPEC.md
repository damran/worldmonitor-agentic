# Gate B-4d — dead-letter retention / pruning

- **Gate:** B-4d (the **dead-letter half** of audit finding **M-6**; the last B-4-family slice, named
  by the B-4c judge). Single-slice gate.
- **Branch:** `gate/b4d-deadletter-pruning` (off `master` @ `f8c322b`; clean).
- **ADR:** `docs/decisions/0053-dead-letter-retention.md` (PROPOSED).
- **Severity:** MEDIUM (M-6).
- **The finding (M-6, `docs/reviews/PRODUCTION_READINESS_AUDIT_2026-06-23.md:183`):**
  "`ingest_dead_letter` has no retention (`task_run` is pruned, but only at startup — H-8). Disk
  eventually fills." The `ingest_dead_letter` table (`src/worldmonitor/db/models.py:78-107`) grows
  **unbounded** — every quarantined ingest/resolve row accretes forever with no GC.
- **Why this is a near-clone (deliberately).** `IngestDriver.prune_task_runs`
  (`src/worldmonitor/runner/driver.py:125-152`) already implements exactly this retention pattern for
  `task_run`: read a `*_retention_days` setting, compute `cutoff = now - timedelta(days=retention)`,
  delete rows older than the cutoff (retention ≤ 0 disables), return the count, log at info. It is
  called once per maintenance cycle (`driver.py:309-310`, alongside `recover_stale()`). B-4d adds the
  symmetric `prune_dead_letters` and wires it in the same cycle.
- **NOT in this gate (hard stops, §10).** M-6 ALSO covers landing-zone orphan GC, disk-usage alerting,
  and deterministic `record.key`. Those are **OUT OF SCOPE** for B-4d — separate follow-ups. This gate
  is **dead-letter retention ONLY**.

## 1. The gap (verified against B's code @ `f8c322b`)

`src/worldmonitor/db/models.py:78-107` — `IngestDeadLetter`:

```
96  __tablename__ = "ingest_dead_letter"
98  id: Mapped[str]            = mapped_column(String(64), primary_key=True)
99  connector_id: Mapped[str]  = mapped_column(String(128))
101 source_key: Mapped[str]    = mapped_column(String)
103 source_record: Mapped[str | None] = mapped_column(String, default=None)
104 stage: Mapped[str]         = mapped_column(String(16), index=True)
105 error: Mapped[str]         = mapped_column(String, default="")
106 created_at: Mapped[datetime] = mapped_column(DateTime(timezone=True), server_default=func.now())
```

Every quarantined record (ingest stages `land`/`map`; resolution stages `resolve-row`/`resolve-batch`/
`resolve-write`/`resolve-noid`/`resolve-incompat`/`signoff-poison` — per the docstring, ADRs 0038 /
0041) is inserted here and **never deleted**. The driver loop runs forever; on a busy or flaky source
the table grows without bound and "disk eventually fills" (M-6). Per the `IngestDeadLetter` docstring,
**all quarantines are replayable** — they are an error-audit trail, not canonical or human-decision
state — so a bounded retention window loses nothing that cannot be regenerated by re-ingesting.

The proven sibling, `prune_task_runs` (`driver.py:125-152`), is the template:

```
132 retention = self._settings.task_run_retention_days
133 if retention <= 0:
134     return 0
135 cutoff = (now or datetime.now(UTC)) - timedelta(days=retention)
136 with self._sessions() as session:
137     stale_ids = list(session.execute(
138         select(TaskRun.id).where(
140             TaskRun.status.in_(("ok", "error")),
141             TaskRun.finished_at.is_not(None),
142             TaskRun.finished_at < cutoff,
143         )).scalars())
146     if stale_ids:
147         session.execute(delete(TaskRun).where(TaskRun.id.in_(stale_ids)))
148         session.commit()
149 deleted = len(stale_ids)
150 if deleted:
151     logger.info("pruned %d finished task_run row(s) older than %dd", deleted, retention)
152 return deleted
```

The **one structural difference**: `task_run` rows have a lifecycle (`running` → `ok`/`error`), so
`prune_task_runs` filters on `status in (ok, error)` AND `finished_at is not null` AND
`finished_at < cutoff` to avoid deleting in-flight runs. **Dead-letters are terminal** — an
`IngestDeadLetter` row is written once and never mutated; there is no "in-flight" dead-letter. So
`prune_dead_letters` prunes ALL rows with `created_at < cutoff` — **no status / finished_at filter**.

## 2. The load-bearing testable invariant

One positive invariant, failing-test-first:

> `IngestDriver.prune_dead_letters(*, now=None) -> int` deletes every `IngestDeadLetter` row whose
> `created_at < (now - dead_letter_retention_days)`, **keeps** every row inside the window, treats
> `retention <= 0` as **disabled** (deletes nothing, returns 0), and returns the count deleted.

Determinism: the test injects `now` (mirroring the `prune_task_runs` test) and seeds rows with
`created_at` straddling the cutoff, so the cutoff arithmetic is exact and reproducible — no wall clock.

The **negative-space invariant** (FROZEN, §9): the driver's ingest/resolve cadence, serialization
(`_resolve_lock`), `recover_stale`, `prune_task_runs`, and `_finalize` behaviour are **unchanged**.
B-4d is strictly **additive** — a new method plus one new call in the same maintenance cycle.

### 2.1 How the invariant is proven (deterministic, non-flaky)

| Concern | How proven |
|---|---|
| Rows older than the window are deleted | **integration** — `test_ingest_driver.py`, seed a row with `created_at = _NOW - 60d` (window 30d), assert pruned |
| Rows inside the window are kept | **integration** — same test, seed a row with `created_at = _NOW`, assert retained |
| `retention <= 0` disables (deletes nothing) | **integration** — same suite, settings retention 0, seed an old row, assert it survives + return 0 |
| Returns the deleted count | **integration** — assert the return value equals the number of straddling-old rows |

Reuses the EXISTING `test_ingest_driver.py` harness (`_harness`, `_NOW`, `sessions`, the `postgres_dsn`
+ `clean_graph` testcontainers fixtures) — the same harness the `prune_task_runs` test uses. Docker is
available locally; run the suite locally for this gate (do not defer to CI-only).

## 3. Design — smallest correct footprint

Four files. No DB model change, no migration.

### 3.1 `settings.py` — one new knob (mirror `task_run_retention_days`)

Add directly under `task_run_retention_days` (`settings.py:82`):

```python
# Dead-letter (ingest_dead_letter) rows older than this are pruned each driver maintenance
# cycle so the replayable error-audit table does not grow without bound (M-6 / ADR 0053). 0 disables.
dead_letter_retention_days: int = Field(default=30, ge=0)
```

- **Type / bound / env var:** `int`, `ge=0`, env `DEAD_LETTER_RETENTION_DAYS` — byte-for-byte the same
  shape as `task_run_retention_days` (pydantic-settings derives the env var from the field name).
- **Default = 30 days — justified.** This **matches the house value** (`task_run_retention_days`'s
  default is `30`, `settings.py:82`). Dead-letters are a *replayable audit trail* (the `IngestDeadLetter`
  docstring: "All quarantines are replayable"): the window must be generous enough to investigate a
  failure spree after the fact, but bounded so disk does not fill. 30 days covers a typical incident
  review cycle and aligns with the existing run-history retention — one mental model, one default, for
  both bounded ops tables. Documented in ADR 0053.

### 3.2 `driver.py` — `prune_dead_letters` (mirror `prune_task_runs`)

Add `IngestDeadLetter` to the existing model import (`driver.py:42`):

```python
from worldmonitor.db.models import ConnectorInstance, ErQueueItem, IngestDeadLetter, TaskRun
```

(`delete`, `select`, `UTC`, `datetime`, `timedelta`, `logger` are already imported — `driver.py:37`,
`:34`.) Add the method immediately after `prune_task_runs` (after `driver.py:152`):

```python
def prune_dead_letters(self, *, now: datetime | None = None) -> int:
    """Delete ingest_dead_letter rows older than the retention window.

    Keeps the replayable dead-letter (error-audit) table bounded
    (``DEAD_LETTER_RETENTION_DAYS``, M-6 / ADR 0053). Dead-letters are terminal —
    there is no in-flight state — so ALL rows older than the cutoff are pruned (no
    status/finished_at filter, unlike :meth:`prune_task_runs`). Retention of 0
    disables it. Returns the number deleted.
    """
    retention = self._settings.dead_letter_retention_days
    if retention <= 0:
        return 0
    cutoff = (now or datetime.now(UTC)) - timedelta(days=retention)
    with self._sessions() as session:
        stale_ids = list(
            session.execute(
                select(IngestDeadLetter.id).where(IngestDeadLetter.created_at < cutoff)
            ).scalars()
        )
        if stale_ids:
            session.execute(delete(IngestDeadLetter).where(IngestDeadLetter.id.in_(stale_ids)))
            session.commit()
    deleted = len(stale_ids)
    if deleted:
        logger.info("pruned %d dead_letter row(s) older than %dd", deleted, retention)
    return deleted
```

Shape is identical to `prune_task_runs` except: (a) the setting read, (b) NO status/finished_at filter
(terminal rows), (c) the log message. The `select id` → `delete in ids` two-step is kept verbatim
(mirrors the proven method; portable across the testcontainers postgres + any backend).

### 3.3 `driver.py` — wire it into the SAME maintenance cycle

`run_forever` runs the startup maintenance at `driver.py:309-310`:

```python

self.prune_task_runs()
self.prune_dead_letters()   # <-- add this one line, right after prune_task_runs
```

It runs once at startup in the same cycle as `prune_task_runs` (same cadence, same place). This is the
minimal, additive wiring — it does NOT touch the tick loop, `run_due_ingests`, `run_resolution`, the
`_resolve_lock` serialization, the heartbeat, `_finalize`, or `prune_task_runs` itself.

> **Note on cadence.** `prune_task_runs` today runs **once at startup** (H-8 noted it is startup-only).
> B-4d mirrors that placement exactly — it does NOT "fix" the startup-only cadence (that is H-8's
> concern, not this gate). Putting `prune_dead_letters` anywhere other than next to `prune_task_runs`
> would expand the gate. If a future gate moves both to a periodic in-loop cycle, both move together.

### 3.4 `.env.example` — document the knob

## 6. Migration conclusion

**NONE.** The prune filters on `created_at`, which **already exists** on `ingest_dead_letter`
(`models.py:106`). No column, table, constraint, or index is added; `db/models.py` is **untouched**.

`prune_task_runs` queries `finished_at` with **no dedicated index** and is the accepted pattern; a
periodic full-scan delete on a table that this gate is specifically keeping *small* is fine. We
deliberately do **not** add a `created_at` index: that would require an alembic revision, which would
trigger `tests/integration/test_migrations.py` (alembic head == `create_all`, ADR 0030) and grow the
gate for no real benefit on a bounded table. Migration-free is the minimal correct choice; if real-world
volume ever makes the scan hot, a `created_at` index is a clean, separate, single-migration follow-up.

`tests/integration/test_migrations.py` is **NOT triggered and MUST stay green** (FROZEN).

## 7. Person-affecting / sign-off assessment

**Ops retention of an error-audit table — NOT person-affecting. No human sign-off.** `ingest_dead_letter`
holds quarantined *records that failed to ingest/resolve*; per its docstring, **all quarantines are
replayable**. Pruning an expired dead-letter loses **no canonical entity, no merge decision, and no
human-made determination** — the source can be re-ingested to regenerate it. This is the explicit
contrast with the H-1 reject-durability concern, which lives in **`resolver_judgement`** (a durable
record of a *human reject decision*) — that table is NOT touched here and is NOT subject to this
retention. `dead_letter_retention_days` is an operations/disk knob, not an individual-affecting score or
ER threshold, so the "changes affecting a real person always need human sign-off" rule does **not**
apply. `human_fork: false` — determinable from CLAUDE.md + the code; no OPEN architectural question.
ADR 0053 is PROPOSED, no human STOP.

## 8. Locked invariants the gate must hold + APPROVE/DENY

- **G1 — provenance on every node AND edge.** PRESERVED **VACUOUSLY** — this gate deletes expired
  *dead-letter* rows; it writes no node/edge, touches no `map()`, no provenance stamp, no graph writer.
  **DENY** if any graph-write or provenance path is touched, or any G1 test regresses.
- **Append-only / canonical-canonical only via the guard.** PRESERVED **VACUOUSLY** — no resolver, no
  merge, no threshold, no graph mutation, no merge-guard touch. The append-only invariant governs the
  **canonical graph / ledger**; `ingest_dead_letter` is an ops error-audit table, explicitly replayable,
  and bounding it is the designed-for GC. **DENY** if any merge / threshold / guard / writer / ledger
  code changes, or if the prune is pointed at any table other than `ingest_dead_letter`.
- **The gate's OWN positive invariant (load-bearing).** `prune_dead_letters` deletes rows older than the
  window, keeps rows inside it, `retention <= 0` disables, returns the count. **DENY** if it deletes
  in-window rows, fails to disable at `retention <= 0`, deletes from the wrong table, or filters on the
  wrong column.
- **Negative-space invariant (FROZEN).** Driver ingest/resolve cadence, serialization (`_resolve_lock`),
  `recover_stale`, `prune_task_runs`, `_finalize`, the heartbeat — all **unchanged** (purely additive).
  **DENY** if any of those behaviours changes or its test is weakened.

**APPROVE** iff: `prune_dead_letters` meets the §2 invariant under the §5 test; it is wired once after
`prune_task_runs` in the same cycle; the setting + `.env.example` doc exist with default 30; no
migration / no `db/models.py` change; the FROZEN driver-behaviour suites stay green. **DENY** on any
cadence/serialization/recover/`prune_task_runs` regression, any wrong-table/wrong-column prune, any
migration, or any FROZEN regression.
