# Runbook — database migrations (ADR 0030 + ADR 0084)

> **Audience:** an operator running WorldMonitor on the persistent single-node host (D1, ADR 0042).
> This runbook covers the **migrate-while-stopped** procedure (the safe default for this
> deployment), the rollback procedure, what the `lock_timeout` guard does, and the online-safe
> patterns to follow when writing future migrations.

---

## 1. The M-5 `lock_timeout` guard (what it does and when to disable it)

`migrations/env.py::_run` calls `SET LOCAL lock_timeout = '<N>ms'` on a Postgres connection
**before every migration batch** (ADR 0084).  This guard makes a migration that cannot acquire
its `ACCESS EXCLUSIVE` DDL lock **abort fast** (Postgres error `55P03 LockNotAvailable`) instead
of waiting indefinitely and blocking every subsequent `INSERT`/`UPDATE` on the locked table —
which stalls the driver's enqueue path.

**Default:** `MIGRATION_LOCK_TIMEOUT_MS=3000` (3 seconds).
**Opt-out:** `MIGRATION_LOCK_TIMEOUT_MS=0` disables the guard (Postgres default: no timeout).

> Use `MIGRATION_LOCK_TIMEOUT_MS=0` when running migrations **while the driver is stopped** — no
> live traffic competes for the lock, so there is nothing to abort against and the guard adds no
> value.  The environment variable is read at migration time from the process environment (or
> `.env`), so you can set it for the migration command only without changing a persistent config.

The guard is **Postgres-only**: it is silently skipped on SQLite and any other non-Postgres
dialect.

## 2. Migrate-while-stopped procedure (the safe default)

This is the recommended procedure for all migrations on the single-node deployment.  No
`CONCURRENTLY` / `NOT VALID` patterns are needed because there is no live traffic competing for
the lock.

```bash
# 1. Stop the driver gracefully (lets the current ingest/resolve pass finish).
#    With the bundled compose stack (the deployed shape — there is no systemd unit):
docker compose -f deploy/compose.yaml stop driver
#    Or, if you started it directly:
pkill -SIGTERM -f "worldmonitor.runner.driver"

# 2. Confirm the driver is stopped (no running task_run rows).
psql "$POSTGRES_DSN" -c "SELECT COUNT(*) FROM task_run WHERE status = 'running';"
# Expect: 0

# 3. Run the migration (disable lock_timeout since the DB is idle).
MIGRATION_LOCK_TIMEOUT_MS=0 python -m worldmonitor.db.migrate
# Equivalent alembic CLI:
# MIGRATION_LOCK_TIMEOUT_MS=0 alembic upgrade head

# 4. Verify the migration applied and the schema is consistent.
alembic check                      # raises AutogenerateDiffsDetected on drift
alembic current                    # should show the head revision
# Or run the migration convergence test (Postgres testcontainer required):
uv run pytest tests/integration/test_migrations.py -q

# 5. Restart the driver.
sudo systemctl start worldmonitor-driver
# Or:
python -m worldmonitor.runner.driver &
```

If `alembic upgrade head` fails (e.g., the `lock_timeout` was NOT disabled and another process
held the lock), the migration **aborts and rolls back automatically** (single-transaction DDL).
The schema is left unchanged.  Fix the contention (stop the driver, wait for the lock holder to
finish), then rerun.

## 3. Rollback procedure

Alembic migrations are single-transaction: a mid-migration failure automatically rolls back all
DDL in that transaction.  Manual rollback is therefore only needed when:

- A migration **succeeded** but the new code is broken and you need to revert to the previous
  schema.
- You applied the wrong revision.

```bash
# Roll back the most recently applied migration.
alembic downgrade -1

# Roll back to a specific revision (e.g. 0005_er_gold_pair):
alembic downgrade 0005_er_gold_pair

# Verify:
alembic current   # should show the prior revision
alembic check     # should pass (models and schema agree at the prior revision)
```

> **Before rolling back a migration that added tables or columns that hold data:**
> take a Postgres backup first (`docs/runbooks/backup-restore.md`) — `downgrade` will DROP those
> objects and their data permanently.

If the schema is in an unknown state (e.g., a crash partway through a multi-step migration that
somehow committed partial DDL), compare the live schema against `Base.metadata`:

```python
from sqlalchemy import inspect, create_engine
from worldmonitor.db.engine import _missing_schema_items, make_engine
engine = make_engine("postgresql+psycopg://...")
print(_missing_schema_items(inspect(engine)))
```

An empty list = schema is consistent with the ORM models.  A non-empty list = partial schema.
Restore from backup and rerun `alembic upgrade head` from a clean baseline.

## 4. Online-safe patterns for FUTURE migrations

If you are writing a migration against a large table (e.g., `er_queue_item` with millions of
rows) that will run **without stopping the driver**, use these patterns instead of the plain
alembic helpers.

> **IMPORTANT:** `CREATE INDEX CONCURRENTLY` and the `NOT VALID` + `VALIDATE` pattern require
> **autocommit** (they cannot run inside a transaction).  `env.py` currently wraps all
> migrations in one transaction (`context.begin_transaction()`), which is incompatible.
>
> Before writing a `CONCURRENTLY` migration, **enable `transaction_per_migration=True`** in
> `env.py` (set it in `context.configure(...)`) so each migration script runs in its own
> isolated transaction.  This is **DEFERRED** in ADR 0084 D1-DEFERRED until the first migration
> that actually needs it.  Record that ADR decision when you enable it.

### 4a — New index on a large table

```python
# BAD: takes ACCESS EXCLUSIVE for the full index build — blocks all writers.
op.create_index("ix_foo_bar", "foo", ["bar"])

# GOOD: CONCURRENTLY acquires only ShareUpdateExclusiveLock — writers continue.
# Requires autocommit / transaction_per_migration=True (see note above).
op.execute("CREATE INDEX CONCURRENTLY IF NOT EXISTS ix_foo_bar ON foo (bar)")
```

`CREATE INDEX CONCURRENTLY` is slower (two table scans instead of one) but safe for large live
tables.

### 4b — New constraint on a large table

```python
# BAD: full table scan to validate while holding ACCESS EXCLUSIVE.
op.create_unique_constraint("uq_foo_bar", "foo", ["bar"])

# GOOD: add NOT VALID (catalog update only, no scan, very brief lock) then
# validate in a separate step (ShareUpdateExclusiveLock, no blocking writers).
# Requires autocommit / transaction_per_migration=True (see note above).
op.execute(
    "ALTER TABLE foo ADD CONSTRAINT uq_foo_bar UNIQUE (bar) NOT VALID"
)
# In a SEPARATE migration (or a separate autocommit step):
op.execute("ALTER TABLE foo VALIDATE CONSTRAINT uq_foo_bar")
```

### 4c — Adding a NOT NULL column

```python
# GOOD for small tables: direct add.
op.add_column("foo", sa.Column("bar", sa.String(), nullable=False, server_default=""))

# GOOD for large tables (Postgres ≥ 11 with a stable server_default):
# Postgres 11+ stores the default in the catalog for NOT NULL + server_default columns
# without a full-table rewrite. Still takes ACCESS EXCLUSIVE briefly (catalog update).
# If the rewrite cost is unacceptable, use the multi-step: add nullable, backfill,
# add NOT NULL — each step in its own transaction.
```

### 4d — `lock_timeout` in the migration session

When `MIGRATION_LOCK_TIMEOUT_MS` is set (non-zero, the default), the guard aborts the migration
if it cannot acquire the `ACCESS EXCLUSIVE` lock within the timeout.  For an online-safe migration:

1. Run it during low-traffic hours.
2. If it times out, rerun — typically the window between bursts is enough.
3. Or stop the driver, set `MIGRATION_LOCK_TIMEOUT_MS=0`, and use the migrate-while-stopped
   procedure above.

## 5. Quick reference

| Task | Command |
|---|---|
| Apply all pending migrations | `python -m worldmonitor.db.migrate` |
| Apply with lock_timeout disabled | `MIGRATION_LOCK_TIMEOUT_MS=0 python -m worldmonitor.db.migrate` |
| Check schema consistency | `alembic check` (raises on any drift) |
| Show current revision | `alembic current` |
| Show migration history | `alembic history` |
| Roll back one step | `alembic downgrade -1` |
| Roll back to specific revision | `alembic downgrade <revision_id>` |
| Generate a new migration | `alembic revision --autogenerate -m "<description>"` |
